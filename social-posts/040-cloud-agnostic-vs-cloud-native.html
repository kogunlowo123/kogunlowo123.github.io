The Real Trade-offs

  • Kubernetes is the poster child for cloud-agnostic — but EKS, AKS, and GKE each have proprietary integrations for IAM, networking, and storage that make "portable" K8s a myth.
  • Cloud-native services like Lambda and DynamoDB reduce operational burden by 80% — but your code becomes deeply coupled to AWS SDKs and event formats.
  • Terraform provides multi-cloud IaC, but each provider's resource model is different. Switching clouds means rewriting 70% of your Terraform anyway.
  • Cloud-agnostic means giving up Azure Functions bindings, Cloud Run auto-scaling, and Lambda@Edge — the features that make cloud compelling in the first place.
  • The honest math: most companies will never switch clouds. The cost of building for portability exceeds the cost of a theoretical migration that never happens.
  • The pragmatic approach: use cloud-native for differentiated workloads, Kubernetes for commodity workloads. Keep your data layer portable with standard protocols (PostgreSQL, S3-compatible storage).
