Feature Flags vs Feature Branches: You're Solving Different Problems

Teams adopt feature flags thinking they replace feature branches. They don't. Branches isolate code during development. Flags isolate features during deployment. One lives in Git, the other lives in production. Confusing them creates technical debt in both systems.

The Real Trade-offs

  • Feature flag debt is worse than branch debt: Stale branches are visible and deletable. Stale feature flags hide in production code, create untested paths, and multiply combinatorial complexity. A 500-flag codebase is unmaintainable.
  • Flags enable progressive rollout branches can't: Roll out to 5% of users, monitor error rates, expand to 50%, then 100%. No branching strategy provides this runtime safety net.
  • The testing matrix explodes: Every active feature flag doubles your test state space. Two flags means 4 combinations, ten flags means 1024. Teams rarely test all combinations and ship bugs in untested flag states.
  • Flag services add runtime dependency: LaunchDarkly down means your feature flags freeze in their last state. You've added a critical dependency for the privilege of if-statements in production.
  • The disciplined approach is both: Short-lived feature branches for code review, feature flags for deployment control. Delete flags within 2 weeks of full rollout. Treat flag cleanup as mandatory, not aspirational.
