InfluxDB's Licensing Killed Its Community

InfluxDB removed clustering from the open-source version, switched to a custom query language (Flux), then pivoted again to InfluxDB 3.0 with SQL. Meanwhile TimescaleDB sits on PostgreSQL with full SQL support, extensions, and a thriving ecosystem. Licensing decisions have consequences.

The Real Trade-offs

  • SQL vs. custom query languages: TimescaleDB uses standard SQL. InfluxDB went from InfluxQL to Flux to now SQL in v3.0. Teams that invested in Flux are left with migration debt.
  • PostgreSQL ecosystem leverage: TimescaleDB inherits JOINs, triggers, foreign data wrappers, pgvector, and every PostgreSQL tool. InfluxDB is an island with its own tooling.
  • Write performance at extreme scale: InfluxDB's columnar storage (IOx engine in v3) can outperform TimescaleDB for pure time-series write-heavy workloads. Purpose-built has advantages.
  • Community trust matters: Removing clustering from OSS, deprecating query languages, and multiple architectural pivots eroded InfluxDB's community trust. TimescaleDB's roadmap has been more predictable.
  • Operational simplicity: If you already run PostgreSQL, TimescaleDB is just an extension — no new database to operate, monitor, backup, and secure. That operational saving is real.
