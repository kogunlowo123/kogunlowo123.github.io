Lakehouse Killed the Data Lake vs Warehouse Debate

Databricks coined "lakehouse" and now every vendor claims to be one. ACID transactions on object storage, SQL query engines over Parquet files, and unified batch+streaming. The lakehouse pattern merges the best of both worlds — but is it just a data lake with table formats bolted on?

The Real Trade-offs

  • Lakehouse is real, not just marketing: ACID transactions, time travel, schema evolution, and partition pruning on object storage genuinely close the gap with traditional warehouses.
  • Performance still lags dedicated warehouses: For complex analytical queries, Snowflake and BigQuery still outperform lakehouse query engines. The gap is closing but not yet closed.
  • Open table formats are the real win: Iceberg, Delta, and Hudi mean your data isn't locked into a vendor's proprietary format. Query it with Spark, Trino, Flink, or any compatible engine.
  • Operational complexity increases: A lakehouse requires managing table formats, compaction, metadata, and multiple query engines. A managed warehouse handles all of this for you.
  • Cost optimization is the compelling case: Storing data on S3 at $0.023/GB vs. warehouse storage at $20+/TB/month adds up at scale. Compute-storage separation lets you scale independently.
