dbt vs Stored Procedures — The SQL Transformation War

dbt brought version control, testing, and documentation to SQL transformations — things stored procedures never had. But stored procedures execute inside the database engine with zero data movement, handle procedural logic natively, and have been battle-tested for 30 years. The dbt hype dismisses stored procedures as legacy. The enterprise reality is that they work and they're fast.

The Real Trade-offs

  • Version control changes everything: dbt models live in Git. You get pull requests, code review, rollbacks, and branching for SQL transformations. Stored procedures typically live in the database with manual change tracking. This alone justifies dbt for many teams.
  • Testing and documentation are first-class: dbt's schema tests (unique, not_null, accepted_values, relationships) and auto-generated DAG documentation solve problems that stored procedures leave to manual discipline. Most stored procedure codebases have zero tests.
  • Stored procedures handle procedural logic better: Loops, cursors, conditional branching, error handling with TRY/CATCH — some transformations are inherently procedural. dbt's SELECT-only paradigm requires workarounds (macros, run_query) that feel hacky for complex ETL logic.
  • Performance edge goes to stored procedures: They execute inside the database engine, can use temp tables efficiently, and benefit from query plan caching. dbt compiles to SQL that may not optimize as well for complex multi-step transformations.
  • The pragmatic answer is both: Use dbt for the analytics layer — staging, intermediate, mart models where testability and documentation matter most. Keep stored procedures for complex ETL logic that's procedural by nature. Don't rewrite working stored procedures just to say you use dbt.
