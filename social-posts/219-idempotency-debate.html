The Real Trade-offs

  • Idempotency keys need TTLs: Storing every idempotency key forever is a storage leak. Set TTLs (24-72 hours typical) — but make them long enough to cover retry windows and network partition recovery.
  • DynamoDB conditional writes are perfect for this: Use PutItem with a condition expression (attribute_not_exists). If the key exists, the write fails atomically — no race conditions.
  • The response caching problem: When a duplicate request arrives, you must return the same response as the original. This means storing the full response payload alongside the idempotency key.
  • Payment processing makes this non-negotiable: Charging a customer twice because of a network timeout retry is not a bug — it is a business-ending event. Every payment endpoint must be idempotent.
  • Natural idempotency is better than forced: PUT /orders/123 with the full state is naturally idempotent. POST /orders with a unique ID is not. Design APIs to be idempotent by nature where possible.
