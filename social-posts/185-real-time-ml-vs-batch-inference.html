Real-Time ML vs Batch Inference — When Milliseconds Actually Matter

Every startup wants "real-time ML predictions." Most of them are serving recommendations that could be pre-computed hourly with zero user-perceived difference. Real-time inference requires always-on GPU endpoints, model optimization, and 24/7 monitoring. Batch inference runs on a schedule, costs a fraction, and handles 80% of ML use cases. Know which one you actually need.

The Real Trade-offs

  • Cost difference is 10-50x: A SageMaker real-time endpoint on ml.g5.xlarge runs ~$1.40/hour whether serving 1 request or 1,000. Batch transform on the same instance processes your entire dataset and terminates. For predictions needed once daily, batch is dramatically cheaper.
  • Real-time is essential for interactive experiences: Fraud detection at payment time, content ranking during page load, dynamic pricing during checkout — these require sub-200ms inference. No batch schedule can substitute for predictions that must happen in the request path.
  • Async inference is the overlooked middle ground: SageMaker Async Inference and Vertex AI batch prediction with auto-scaling handle requests within seconds to minutes. For use cases like document processing or image analysis, this eliminates always-on endpoint costs.
  • Model optimization tax applies only to real-time: Quantization, distillation, ONNX conversion, TensorRT optimization — these efforts only matter when latency is a constraint. Batch inference can use the full-precision model without optimization investment.
  • Pre-compute and cache is the pragmatic pattern: Run batch inference on your user base nightly, store predictions in Redis or DynamoDB, serve from cache at sub-millisecond latency. You get real-time serving speed with batch inference costs. This handles most recommendation and scoring use cases.
