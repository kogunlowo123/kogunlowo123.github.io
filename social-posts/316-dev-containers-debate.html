Dev Containers Solve 'Works on My Machine' for Real

A devcontainer.json in your repo means every developer gets the exact same environment — same OS, same toolchain, same dependencies. No more "I'm on Python 3.9" vs "I'm on 3.11" arguments. But is the container startup overhead and Docker dependency worth it for every project?

The Real Trade-offs

  • Reproducibility is genuinely solved: Same container image = same environment everywhere. CI/CD, local dev, and cloud IDEs all run identical toolchains.
  • Docker dependency is a bottleneck: Docker Desktop licensing, resource consumption on laptops, and WSL2 quirks on Windows create friction that nix or asdf avoid.
  • Startup time matters: Building a complex dev container can take 5-10 minutes on first run. Pre-built images help but require maintenance infrastructure.
  • Not every project needs this: A simple Node.js app with zero native dependencies doesn't need a dev container. The overhead isn't justified for trivial projects.
  • GPU and hardware access is tricky: ML engineers needing GPU access, embedded developers needing USB passthrough — dev containers struggle with hardware-dependent workflows.
