Lambda Layers vs Container Images: The Packaging Debate

Lambda Layers were the original way to share dependencies — up to 5 layers, 250MB unzipped total. Then AWS added container image support with a 10GB limit. Layers are simpler but fragile. Containers are heavier but portable. Your team's workflow should decide.

The Real Trade-offs

  • Layers enable code sharing but create coupling: A shared utility layer used by 50 functions means updating that layer requires redeploying all 50 functions. Version pinning helps, but most teams don't pin — leading to "it works on my layer" debugging.
  • Container images give you local parity: Build locally with Docker, test locally with Docker, deploy the same image to Lambda. No more "it works on my machine but not in Lambda" because the execution environment is identical.
  • Cold starts differ significantly: Zip-packaged functions with layers cold start faster because AWS optimizes the extraction path. Container images, especially large ones, take longer to pull and extract — though AWS caches aggressively after first invocation.
  • The 250MB limit is a real constraint: Machine learning models, native binaries, or large dependency trees easily exceed 250MB. Container images give you 10GB — enough for most ML inference workloads without resorting to EFS mounts.
  • CI/CD complexity varies: Layers require separate build-and-publish pipelines, version management, and cross-account sharing via resource policies. Container images use standard Docker workflows — docker build, docker push to ECR, update Lambda.
