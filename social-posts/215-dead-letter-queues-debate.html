DLQs Without Alerting Are Message Graveyards

DLQs without alerting are message graveyards. Dead letter queues collect failed messages silently, and without CloudWatch alarms and automated reprocessing, those messages rot until someone notices — usually your customers first.

The Real Trade-offs

  • Every DLQ needs a CloudWatch alarm: Metric: ApproximateNumberOfMessagesVisible > 0. If this alarm is not set, your DLQ is just a black hole where business-critical messages disappear silently.
  • DLQ redrive is built into SQS now: AWS added native DLQ redrive — move messages back to the source queue with one API call. No more writing custom replay scripts.
  • Retention period is your ticking clock: DLQ messages have the same 14-day max retention. If you do not process or archive them within that window, they are gone forever.
  • Categorize before redriving: Blindly replaying all DLQ messages causes the same failures. Parse error metadata, fix the root cause, then redrive. A categorization Lambda on the DLQ prevents replay storms.
  • Lambda destinations beat DLQs for async: For async Lambda invocations, use Lambda Destinations instead of DLQs — you get the full event payload, error details, and can route to EventBridge for flexible handling.
