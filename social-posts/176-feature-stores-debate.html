Feature Stores Solve a Problem Most Teams Don't Have

Feast, Tecton, SageMaker Feature Store — they solve feature reuse and training-serving skew for teams with 50+ ML models in production. Most companies have 2-3 models and a Pandas script. You don't need a feature store. You need a SQL query and a cron job.

The Real Trade-offs

  • Training-serving skew is a real problem: When features computed differently in training vs. inference, model accuracy drops silently. Feature stores guarantee consistency — but only matters at scale.
  • Feature reuse saves engineering time: At Uber-scale with hundreds of models sharing features, a centralized store prevents duplicate feature engineering. At startup-scale, it's premature optimization.
  • Online serving adds infrastructure: Feature stores need Redis or DynamoDB for real-time serving. That's another system to operate and pay for when a database query might suffice.
  • Point-in-time correctness matters for finance: Feature stores handle time-travel queries to avoid data leakage in training. For financial models, this is critical and hard to implement manually.
  • Start simple, add later: Build features in SQL, store in your warehouse, serve from cache. When you hit genuine reuse and consistency pain, then adopt a feature store. Not before.
