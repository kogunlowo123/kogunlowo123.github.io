PrivateLink vs VPC Peering: Service-Level vs Network-Level Connectivity

VPC Peering connects two VPCs at the network layer — every resource in VPC A can potentially reach every resource in VPC B. PrivateLink exposes a single service through an ENI in the consumer's VPC. One is a highway; the other is a private delivery entrance. Most teams default to peering when they should be using PrivateLink.

The Real Trade-offs

  • Security posture is fundamentally different: VPC Peering creates a bidirectional network path. Even with security groups and NACLs, the blast radius of a compromised host spans both VPCs. PrivateLink is unidirectional — the consumer can reach the service, but the service provider can't initiate connections back.
  • CIDR overlap kills peering: VPC Peering requires non-overlapping CIDRs. In large organizations with hundreds of VPCs, IP address management becomes a nightmare. PrivateLink uses ENIs with IPs from the consumer's VPC — no CIDR conflicts possible.
  • Peering is cheaper for bulk traffic: VPC Peering data transfer in the same AZ is free, and cross-AZ is $0.01/GB. PrivateLink charges $0.01/GB plus $0.01/hr per endpoint. For high-throughput internal traffic, peering costs less.
  • Transitive routing changes the calculus: VPC Peering is non-transitive — if A peers with B and B peers with C, A can't reach C. Transit Gateway solves this but costs $0.05/GB. PrivateLink naturally supports hub-and-spoke because each consumer independently connects.
  • SaaS and multi-tenant architectures need PrivateLink: If you're exposing services to other AWS accounts or customers, PrivateLink is the only production-grade option. You control what's exposed, customers don't see your VPC, and there's no IP conflict risk.
