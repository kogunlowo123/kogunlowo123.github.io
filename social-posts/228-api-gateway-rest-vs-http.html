API Gateway REST vs HTTP API: Which One Should You Actually Use?

AWS gives you two API Gateway flavors and the naming is deliberately confusing. REST API has been around since 2015 with every feature imaginable. HTTP API launched in 2019, costs 71% less, and handles most use cases. Stop paying for features you don't need.

The Real Trade-offs

  • HTTP API is 71% cheaper and faster: At $1.00/M vs $3.50/M requests, HTTP API saves serious money at scale. It also adds ~10ms less latency because it skips the heavier REST API processing pipeline.
  • REST API features you probably don't need: Request validation, API keys with usage plans, caching, WAF integration, resource policies, and request/response transformation. If you're just proxying to Lambda, that's all dead weight.
  • What HTTP API actually lacks: No built-in caching, no WAF integration (put CloudFront in front), no API key management, no request validation, and no edge-optimized endpoints. If you need these, REST API is your only option.
  • JWT authorizers are built into HTTP API: Native JWT authorization without Lambda authorizers. Connect Cognito, Auth0, or any OIDC provider directly. REST API requires Lambda authorizers for the same thing — more code, more cost.
  • The migration trap: REST API and HTTP API have different event formats (v1 vs v2). Your Lambda handler code needs changes. Test thoroughly before switching — the payload structure differences break things silently.
