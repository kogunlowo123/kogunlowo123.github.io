Event-Driven Architecture Is Over-Hyped Complexity

You replaced a simple REST API call with an SNS topic, an SQS queue, a Lambda consumer, and a DLQ. Now debugging requires CloudWatch Logs across 4 services, and you've introduced eventual consistency that your frontend team doesn't understand. Was it worth it?

The Real Trade-offs

  • Event-driven with SQS adds latency and eventual consistency — sometimes you need an immediate response
  • EventBridge is powerful for decoupling, but debugging event flows across 10 services is painful
  • Dead letter queues on SQS become graveyards of unprocessed messages nobody monitors
  • Event-driven shines for SNS fan-out patterns — one event, many consumers
  • A synchronous API Gateway → Lambda → Aurora call is debuggable in minutes, not hours
