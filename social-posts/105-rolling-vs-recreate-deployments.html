Rolling Deployments Hide Bugs in Production

Rolling deployments are the Kubernetes default, but they're sneaky. During a rollout, you have two versions serving traffic simultaneously. If your API changed between versions, some users get v1 responses and others get v2. That's not a deployment — that's a distributed bug.

The Real Trade-offs

  • Version skew is the real problem: During a rolling update, old and new pods coexist. If your new version changes an API contract, response schema, or database schema, users get inconsistent behavior.
  • Recreate causes downtime: All old pods terminate before new ones start. For internal tools this is fine. For customer-facing services, even seconds of downtime trigger SLA violations.
  • Rolling updates need backward compatibility: Every deployment must be N-1 compatible. This is a discipline tax. But it also makes your system more resilient overall.
  • maxSurge and maxUnavailable tuning matters: The defaults are often wrong. Too aggressive and you overwhelm the cluster. Too conservative and deployments take forever.
  • The right answer depends on your service: Stateless API? Rolling is fine. WebSocket server with long connections? Recreate might be cleaner. Database migration? Neither — you need a multi-phase approach.
