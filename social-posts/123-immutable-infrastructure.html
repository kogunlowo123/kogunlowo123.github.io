Immutable Infrastructure Is a Luxury Most Can't Afford

Never SSH into a server. Never patch in place. Every change requires a new image, a new deployment, a new everything. Immutable infrastructure is the ideal — but when your config change takes 45 minutes to bake into an AMI, test, and deploy, you start to question the religion.

The Real Trade-offs

  • Containers made immutability practical: Before containers, immutable meant baking AMIs for 20 minutes. Docker images build in seconds and deploy in milliseconds. Containers are immutable infrastructure for everyone.
  • VM-based immutability is expensive: Baking, testing, and rolling out new AMIs for every config change requires significant automation. Most teams doing VM-based workloads can't justify this investment.
  • Emergency patches break immutability: When a zero-day CVE hits, waiting for a new image pipeline is too slow. Having an escape hatch for in-place patching is practical, not shameful.
  • Configuration drift is the real enemy: Immutability solves drift by making it impossible. If you can't SSH in, you can't create snowflake servers. That guarantee has enormous value.
  • Start with containers, not VMs: If you're pursuing immutability, containerize first. The ROI of immutable containers is 10x that of immutable VMs for most workloads.
