Structured Logs Make Metrics Redundant

If every log line is structured JSON with request duration, status code, and user ID — can't you just derive all your metrics from logs? Tools like Loki and Elasticsearch can aggregate log fields into dashboards. So why maintain two separate pipelines?

The Real Trade-offs

  • Metrics are cheaper at scale: A counter incrementing once per request is bytes. A structured log line per request is kilobytes. At 100k RPS, logs cost 1000x more in storage than equivalent metrics.
  • Log-derived metrics have latency: Querying logs to compute a rate takes seconds. Prometheus counters are pre-aggregated and return in milliseconds. For alerting, this latency gap matters.
  • Logs provide context metrics can't: A metric says "error rate is 5%." A log says "error rate is 5%, and here are the exact request payloads, stack traces, and user IDs." Different tools for different jobs.
  • Unified pipelines reduce complexity: If you can run one pipeline (logs) instead of two (logs + metrics), you have less infrastructure to maintain. For small teams, simplicity wins.
  • The real answer is both: Use metrics for alerting and dashboards. Use logs for debugging. Use traces for request flows. They complement, not replace, each other.
