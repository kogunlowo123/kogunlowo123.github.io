The Real Trade-offs

  • The golden rule: Every alert must be actionable, require human intelligence, and be urgent. If an alert doesn't meet all three criteria, it should be a dashboard widget, not a page.
  • SLO-based alerting changes the game: Instead of alerting on CPU > 80%, alert on "error budget burn rate exceeds threshold." This focuses on user impact, not infrastructure symptoms.
  • Fear of deleting alerts is real: "What if we delete that alert and miss an outage?" Track which alerts get actioned. If an alert hasn't been actioned in 90 days, it's noise.
  • Alert fatigue causes real outages: When on-call engineers are desensitized, they dismiss the one critical alert that matters. The CrowdStrike response delay pattern is well-documented.
  • Start with an alert audit: For every alert, ask: "What do I do when this fires?" If the answer is "check if it auto-resolves" — delete it immediately.
