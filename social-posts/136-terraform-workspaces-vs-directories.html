Terraform Workspaces vs Directories: The Environment Isolation Trap

Terraform workspaces let you run terraform workspace select prod and switch environments. Elegant, right? Until someone forgets which workspace they're in and destroys production. Directory-per-environment is verbose but explicit. In infrastructure, explicit saves careers.

The Real Trade-offs

  • Workspaces hide the blast radius: terraform destroy in the wrong workspace is catastrophic and invisible until it's too late. Directories force you to cd into the right folder — a physical action that creates awareness.
  • Environments are rarely identical: Dev has 1 NAT gateway, prod has 3. Dev uses t3.small, prod uses m5.xlarge. Workspaces push this divergence into conditionals: count = terraform.workspace == "prod" ? 3 : 1. That's code complexity.
  • Directories create code duplication: Without modules, you're copy-pasting HCL across dev/staging/prod. Changes require touching multiple files. Terragrunt solves this but adds another tool to learn.
  • Terraform Cloud changes the equation: TFC workspaces are first-class with separate variables, permissions, and run histories. The workspace anti-pattern in CLI becomes a best practice in TFC.
  • The pragmatic answer is modules + directories: Shared modules in modules/, environment configs in envs/dev/ and envs/prod/ with minimal root configurations. Explicit, DRY, and safe.
