MongoDB Lost the Document Database War to PostgreSQL JSONB

Why run a separate MongoDB cluster when PostgreSQL gives you JSONB with GIN indexes, full ACID transactions, JOINs across relational and document data, and no SSPL licensing headaches? MongoDB solved a problem that PostgreSQL absorbed.

The Real Trade-offs

  • MongoDB's developer experience is superior: The document model, aggregation pipeline, and native driver APIs are purpose-built for document workflows. JSONB queries in PostgreSQL feel bolted on with operator soup (@>, ->>, #>>).
  • Horizontal scaling favors MongoDB: Native sharding with configurable shard keys is built into MongoDB. PostgreSQL horizontal scaling requires Citus, partitioning hacks, or application-level sharding.
  • ACID transactions matter: PostgreSQL's battle-tested transaction model across relational and JSON data in one database eliminates the dual-write problem teams face running MongoDB alongside a relational store.
  • SSPL licensing is a dealbreaker for some: MongoDB's SSPL license means you can't offer it as a service, which is why AWS built DocumentDB. PostgreSQL's permissive license has no such restrictions.
  • Schema flexibility vs. data integrity: MongoDB's schema-less nature is faster for prototyping but leads to schema drift. PostgreSQL's JSON Schema validation and check constraints give structure when you need it.
