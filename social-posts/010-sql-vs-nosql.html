NoSQL Databases Create More Problems Than They Solve

You chose DynamoDB because "it scales infinitely." Now you can't do a simple JOIN, your data model is a nightmare of denormalization, and a single access pattern change requires redesigning the entire table. PostgreSQL on Aurora handles more traffic than 99% of apps will ever see.

The Real Trade-offs

  • Aurora PostgreSQL handles 600K reads/sec with read replicas — more than most apps need
  • DynamoDB requires you to know ALL access patterns upfront — change one and you redesign the table
  • Cloud Spanner proves SQL can scale globally — NoSQL's "SQL doesn't scale" argument is dead
  • NoSQL denormalization means updating a user's name requires writing to 47 different items
  • DocumentDB is just PostgreSQL JSON columns with worse query performance
