Spark Streaming Is Batch Processing in Disguise

Structured Streaming in Spark processes data in micro-batches, not true event-at-a-time streaming. Flink processes each event as it arrives with true low-latency semantics. Calling Spark "real-time" is like calling a very fast bus a taxi — it still waits to fill up before moving.

The Real Trade-offs

  • Latency matters for some, not all: If you need sub-second processing (fraud detection, real-time bidding), Flink's event-at-a-time model is necessary. For dashboards refreshing every 30 seconds, Spark's micro-batch is fine.
  • Spark's unified API is powerful: One framework for batch, streaming, ML, and graph processing. Flink is catching up on batch, but Spark's DataFrame/SQL API is more mature and widely known.
  • Flink's state management is superior: Flink's checkpointing, savepoints, and queryable state are built for long-running streaming jobs. Spark's state management feels bolted on.
  • Talent availability favors Spark: Far more engineers know Spark than Flink. Hiring for Flink expertise is harder and more expensive. This practical reality matters more than technical superiority.
  • Continuous processing mode exists in Spark: Spark added experimental continuous processing mode that processes event-by-event. It's not as mature as Flink, but the gap is closing.
