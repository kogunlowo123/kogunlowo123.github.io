Vector Databases vs Traditional Search — Semantic vs Keyword

The vector database market exploded with the LLM wave — Pinecone, Weaviate, Qdrant, Chroma all raised massive rounds. But Elasticsearch added vector search, PostgreSQL has pgvector, and hybrid search (BM25 + vectors) outperforms pure vector search in most benchmarks. Do you need a dedicated vector DB, or is it a feature of your existing database?

The Real Trade-offs

  • Hybrid search outperforms pure vector: Research consistently shows BM25 keyword search + vector similarity combined (reciprocal rank fusion) outperforms either alone. Elasticsearch, OpenSearch, and Weaviate all support hybrid modes. Pure vector search misses exact-match queries that BM25 handles perfectly.
  • pgvector is good enough for most startups: If you already run PostgreSQL, pgvector adds vector similarity search without new infrastructure. For under 10 million vectors, performance is acceptable. You get ACID transactions, familiar SQL, and no new operational burden.
  • Dedicated vector DBs win at scale: Beyond 100 million vectors with sub-10ms latency requirements, purpose-built databases like Pinecone and Qdrant offer HNSW indexing optimizations, memory-mapped storage, and distributed architectures that bolt-on solutions can't match.
  • Embedding quality matters more than the database: The difference between a good and bad embedding model dwarfs the difference between vector databases. Teams spending weeks choosing Pinecone vs. Weaviate would be better served spending that time on embedding model selection and chunking strategy.
  • Consider your existing stack first: Elasticsearch shop? Use its kNN search. PostgreSQL shop? Start with pgvector. Only adopt a dedicated vector DB when you've outgrown these options or need features like namespace isolation and metadata filtering at scale.
