Ansible vs Terraform for Configuration: Stop Using Both When You Need Neither

Terraform creates the server. Ansible configures it. Sounds clean until you realize immutable infrastructure means you should bake configs into the image and never SSH in again. Are we maintaining two IaC tools for a problem containers already solved?

The Real Trade-offs

  • Terraform's state model doesn't fit config: Terraform tracks infrastructure state brilliantly. But tracking that nginx.conf has line 42 changed is not what a state file is for. Ansible's idempotent tasks are purpose-built for configuration convergence.
  • Ansible still rules legacy environments: Brownfield infrastructure with hundreds of VMs that can't be containerized needs configuration management. Ansible's agentless SSH model works where Terraform never will.
  • Immutable infrastructure eliminates the question: If you never configure a running server — you bake images with Packer and replace instances — then Ansible is unnecessary. But most organizations aren't fully immutable.
  • Cloud-init handles 80% of use cases: For cloud VMs, user data scripts and cloud-init cover package installation and basic configuration. Neither Ansible nor Terraform provisioners needed.
  • The real answer depends on your fleet: Containers? Skip both for config. Cloud VMs with immutable deploys? Terraform + Packer. Legacy VMs needing ongoing config? Terraform + Ansible. Match tools to reality, not ideology.
