Single-Table DynamoDB Design Is an Anti-Pattern

Unpopular opinion: Single-table design in DynamoDB is Rick Houlihan conference theater. It creates write-once, read-never access patterns that only the original developer understands. Multiple tables with clear schemas are easier to maintain, debug, and extend.

The Real Trade-offs

  • Developer onboarding cost: Single-table designs require intimate knowledge of every access pattern encoded in PK/SK schemas. New team members face a wall of cryptic composite keys like "USER#123#ORDER#456".
  • Performance at scale is real: When done right, single-table design reduces round trips and enables collection fetches in one query. For high-throughput APIs, this efficiency matters.
  • Evolution is painful: Adding new access patterns to a single table often means backfilling data, adding GSIs (max 20), or creating materialized views. Multi-table designs add a new table and move on.
  • Cost optimization differs: Single-table shares provisioned capacity across entities, which can be efficient. But hot partitions in one entity affect all entities in the same table.
  • The middle ground exists: Use single-table for genuinely related entities that you query together (user + orders), and separate tables for independent domains. Dogma in either direction hurts.
