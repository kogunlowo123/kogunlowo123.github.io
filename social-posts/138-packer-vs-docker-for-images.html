Packer vs Docker for Image Building: The Immutability Spectrum

Packer builds 20 GB AMIs in 15 minutes. Docker builds 200 MB images in 30 seconds. Both deliver immutable artifacts. But one targets VMs that boot in 60 seconds, the other targets containers that start in milliseconds. Your deployment model determines which matters.

The Real Trade-offs

  • Packer is for the VM world that still exists: GPU workloads, Windows servers, legacy apps, compliance-required AMIs — these run on VMs. Packer builds golden images with security hardening that containers can't replicate.
  • Docker's layer caching is transformative: Change one line of code and only rebuild one layer. Packer rebuilds the entire image from scratch (or from a base). For rapid iteration, Docker wins by orders of magnitude.
  • Security scanning differs fundamentally: Container images have well-understood layer scanning (Trivy, Grype). AMI scanning requires booting instances and running agents. The container security ecosystem is years ahead.
  • Multi-cloud portability favors Docker: A Docker image runs anywhere with a container runtime. An AMI is AWS-only. Packer can build for multiple clouds, but each output is cloud-specific. OCI images are truly portable.
  • The hybrid is common and correct: Packer builds base AMIs for Kubernetes nodes (OS patches, kernel tuning, containerd). Docker builds application images that run on those nodes. They're complementary, not competing.
