Cold Start Strategies: The Serverless Performance Debate

A Java Lambda with a VPC attachment can take 10+ seconds to cold start. Provisioned concurrency fixes it but costs money 24/7. SnapStart promises the best of both worlds. Warm-up pings are a hack that sort of works. Every strategy has a catch — let's break them down.

Strategy Comparison

  • Provisioned Concurrency is expensive insurance: You're paying for always-on Lambda instances — defeating the pay-per-use model. At $0.015/GB-hr, keeping 10 instances warm at 1GB costs ~$108/month. It works perfectly, but the bill is predictable and non-trivial.
  • SnapStart changes the Java cold start game: Snapshots the initialized execution environment after the init phase. Restores in ~200ms instead of 5-10 seconds. But it's Java-only, requires careful handling of unique state, and doesn't work with provisioned concurrency.
  • Scheduled warm-ups are a hack with limits: An EventBridge rule pinging your Lambda every 5 minutes keeps one instance warm. But it only warms ONE concurrent instance. A burst of 50 simultaneous requests means 49 cold starts.
  • Runtime choice is the biggest lever: Python and Node.js cold start in 100-300ms. Java and .NET cold start in 1-10 seconds. Switching from Java to Node.js often eliminates the cold start problem entirely — no provisioned concurrency needed.
  • The VPC cold start myth (now solved): Lambda VPC cold starts used to add 8-10 seconds for ENI attachment. Since AWS introduced Hyperplane in 2019, VPC-attached Lambdas cold start in under 1 second. If you're still avoiding VPC Lambdas for this reason, update your mental model.
