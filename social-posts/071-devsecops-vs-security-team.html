DevSecOps Doesn't Mean Developers Own Security

Security teams rebranded as "DevSecOps" and told developers to fix their own vulnerabilities. Developers aren't trained in security, don't have time for security, and resent being blamed for security. DevSecOps should mean security teams build developer-friendly guardrails — not that they abdicate responsibility.

The Real Trade-offs

  • Security champions bridge the gap: Embed security-minded developers in each team. They're not security engineers — they're developers who champion security practices and escalate complex issues.
  • Paved paths > policies: Instead of saying "don't use eval()", provide a secure library that handles the use case. Developers follow the path of least resistance. Make security the easy choice.
  • Security teams must write code: If your security team can't write a CI pipeline, a Semgrep rule, or a Terraform module, they can't build developer-friendly guardrails. Security engineers need engineering skills.
  • Developers should understand basics: OWASP Top 10, input validation, and authentication patterns are software engineering fundamentals. "I'm not a security person" is not an acceptable excuse.
  • Shared responsibility needs clear ownership: Who owns fixing a critical CVE in a dependency? The developer, the security team, or the platform team? Without clear RACI, vulnerabilities rot in backlogs.
