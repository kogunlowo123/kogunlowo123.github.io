Infrastructure Drift Is a Feature, Not a Bug

Hot take: Not all drift is bad. When an auto-scaling group adds instances, that's drift. When a CDN adjusts edge configurations, that's drift. When a DBA tunes a parameter during an incident, that's drift. Trying to eliminate all drift means fighting against the dynamic nature of cloud infrastructure.

The Real Trade-offs

  • Intentional vs accidental drift: Auto-scaling, self-healing, and dynamic config are intentional drift — the system adapting. Console clicks and hotfixes are accidental drift — process breakdown. Treat them differently.
  • ignore_changes is not a hack: Using lifecycle ignore_changes for ASG desired_count or RDS parameters is legitimate. Some attributes are meant to be managed dynamically, not by Terraform.
  • Drift alerts without remediation create noise: Knowing about drift is useless if you never fix it. Either auto-remediate or accept the drift. Alerting on things you won't act on is just alert fatigue.
  • Zero-drift is a noble but impractical goal: In a large cloud environment, some drift is unavoidable. The question is which resources must be drift-free (security groups, IAM) and which can tolerate drift (scaling params).
  • Continuous reconciliation is the future: Tools like Crossplane continuously reconcile desired vs actual state. This Kubernetes-style approach makes drift self-healing instead of just self-detecting.
