Active-Active is a Myth — Most Teams Run Active-Passive

Everyone draws active-active on their architecture diagrams. But look under the hood: one region handles 95% of traffic, the database has a single writer, and failover is still manual. That's active-passive with extra infrastructure bills.

The Real Trade-offs

  • The database always breaks the dream: True active-active requires multi-writer databases. Aurora Global Database still has a single write region. You're not active-active if writes go to one place.
  • Conflict resolution is the hardest problem: DynamoDB Global Tables support multi-region writes, but last-writer-wins semantics can silently lose data. CRDTs sound great until you implement them.
  • Session state kills active-active: If a user's session is pinned to Region A and it fails, Region B knows nothing about that session. Stateless architectures fix this but most apps aren't truly stateless.
  • Active-passive is honest architecture: Accepting you have a primary region with fast failover is better than pretending you're active-active. It's simpler to operate and cheaper to run.
  • Counter-argument — latency matters globally: For truly global user bases, active-active with edge computing and regional data stores delivers a fundamentally better experience that active-passive can never match.
