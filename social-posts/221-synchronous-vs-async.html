Default to Async — Synchronous Is the Exception

Default to async — synchronous is the exception. Every synchronous HTTP call between services is a latency chain, a failure dependency, and a scaling bottleneck. Async decouples your services and lets each one fail independently.

The Real Trade-offs

  • Latency compounds in sync chains: Service A (50ms) calls B (100ms) calls C (200ms) = 350ms minimum. Async: Service A publishes event (5ms), returns immediately. Total user-facing latency: 55ms.
  • Failure propagation is the real killer: In sync chains, Service C being down means A and B also fail. With async, A succeeds, the message queues, and C processes when it recovers.
  • Async adds complexity in debugging: Distributed tracing across async boundaries is harder. Correlation IDs must flow through queues, events, and consumers. Without them, debugging is impossible.
  • Some operations must be synchronous: User login, payment authorization, real-time search — these need immediate responses. The trick is making only the critical path synchronous and everything else async.
  • Event-driven architecture is async at scale: EventBridge, SNS, SQS form the async backbone. Services publish events, subscribers react independently. No service knows about any other service.
