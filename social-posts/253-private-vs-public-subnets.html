Everything Should Be in Private Subnets — No Exceptions

If your EC2 instance has a public IP, you've already failed. Public subnets should contain exactly two things: load balancers and NAT gateways. Everything else belongs behind a wall. Fight me.

The Real Trade-offs

  • Public IPs are attack surface: Every public IP is a potential target. Security groups help, but defense in depth means not exposing resources to the internet at all. Private subnets eliminate an entire category of attacks.
  • "But I need SSH access": No you don't. AWS Systems Manager Session Manager provides shell access without SSH, without bastion hosts, without public IPs. It's auditable and IAM-controlled.
  • NAT Gateway cost is the counterargument: Yes, NAT Gateways cost money ($0.045/hr + $0.045/GB). For high-egress workloads, this adds up. But the security tradeoff is worth it for most teams — and VPC endpoints eliminate NAT costs for AWS API traffic.
  • Startups push back — and they're wrong: "We're small, security isn't a priority yet." That's exactly when you should build the habit. Private subnets are not harder to set up — they just require a NAT gateway and proper routing.
  • The exception: edge cases exist: Some workloads genuinely need public IPs — game servers, SIP endpoints, certain IoT patterns. But those are 1% of use cases, not the default.
