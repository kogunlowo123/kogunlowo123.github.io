IPv6 Migration: Cost Savings or Operational Headache?

AWS started charging $0.005/hr per public IPv4 address in February 2024 — that's $3.65/month per IP, or $43.80/year. An organization with 1,000 public IPs faces $43,800/year in new charges. IPv6 is free. But the migration path is riddled with compatibility issues and operational complexity.

The Real Trade-offs

  • The cost math is compelling but nuanced: If you have 100 Elastic IPs, you're paying $4,380/year just for addresses. But eliminating public IPv4 doesn't just mean adding IPv6 — you need NAT64/DNS64 for IPv4-only endpoints, dual-stack load balancers, and updated security groups. Engineering time has a cost too.
  • Dual-stack is the safe migration path: Go IPv6-only on day one and you'll break third-party integrations, legacy APIs, and monitoring tools that don't support IPv6. Dual-stack lets you run both protocols, verify compatibility, and gradually shift. The downside: you still pay for IPv4 during transition.
  • IPv6-only subnets for internal workloads: Backend services that only talk to other internal services or AWS APIs can go IPv6-only today. ECS tasks, Lambda functions, and internal microservices are prime candidates. NAT64 handles the rare IPv4-only dependency.
  • Security tooling gaps are real: Some WAFs, IDS/IPS systems, and SIEM tools have incomplete IPv6 support. Security group rules need auditing — IPv6 CIDRs are separate from IPv4 rules. Organizations that skip the security review during migration create blind spots.
  • Start with new workloads, not migrations: The lowest-risk approach: all new VPCs get dual-stack from day one. New ECS services use IPv6-only subnets. Existing production workloads migrate last. This prevents the big-bang risk while capturing savings on new infrastructure.
