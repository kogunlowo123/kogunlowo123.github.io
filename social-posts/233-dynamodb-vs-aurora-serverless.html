DynamoDB vs Aurora Serverless: The Serverless Database Debate

DynamoDB scales to millions of requests per second with single-digit millisecond latency and zero connection management. Aurora Serverless v2 gives you PostgreSQL/MySQL compatibility with auto-scaling ACUs. The choice isn't NoSQL vs SQL — it's access pattern predictability vs query flexibility.

The Real Trade-offs

  • DynamoDB is truly connectionless: Every request is an HTTP call. Lambda handles 1000 concurrent executions? No problem — 1000 simultaneous DynamoDB calls work fine. Aurora Serverless with 1000 Lambda connections? You'll exhaust the connection pool and need RDS Proxy at extra cost.
  • Aurora Serverless v2 doesn't scale to zero: Minimum is 0.5 ACU (~$43/month). DynamoDB on-demand with zero traffic costs literally $0. For sporadic workloads, this matters. Aurora Serverless v1 could scale to zero but was deprecated.
  • Single-table design is DynamoDB's burden: To avoid multiple queries, you model all entities in one table with composite keys. It's powerful but requires upfront access pattern knowledge. With Aurora, you write SQL joins — flexibility you'll appreciate when requirements change.
  • DynamoDB Streams vs Aurora CDC: DynamoDB Streams natively trigger Lambda on item changes — perfect for event-driven architectures. Aurora requires setting up CDC with DMS or polling — more infrastructure, more latency.
  • Cost at scale tells a different story: DynamoDB on-demand at 100M reads/month costs ~$125. Aurora Serverless v2 at equivalent throughput costs $200-400. But if your access patterns need complex joins, DynamoDB forces multiple queries that actually cost more.
