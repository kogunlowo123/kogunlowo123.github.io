WebSockets in Serverless: Fighting the Architecture

WebSockets in serverless is fighting the architecture. You're trying to maintain stateful, long-lived connections on infrastructure designed to be stateless and ephemeral. Use a managed real-time service instead.

The Real Trade-offs

  • Connection management is your problem: API Gateway WebSocket APIs manage the connection, but you must store connection IDs in DynamoDB, handle disconnects, and broadcast manually via the Management API.
  • IoT Core is the serverless WebSocket hack: AWS IoT Core handles millions of persistent connections natively. Use it for real-time even if you're not doing IoT — it's cheaper and more scalable.
  • AppSync subscriptions are the easiest path: If you're already using GraphQL, AppSync handles WebSocket connections, fan-out, and cleanup automatically. Zero connection management code.
  • Long polling is underrated: For update frequencies under 1/second, long polling with API Gateway HTTP API is simpler, cheaper, and more reliable than WebSocket infrastructure.
  • The 2-hour idle timeout is brutal: API Gateway WebSocket connections close after 10 minutes idle (2 hours max). You need heartbeat logic, reconnection handling, and message replay — all on Lambda.
