Why Enterprise AI Agents Need a Curriculum

The gap between AI agent demos and production-grade enterprise agents is enormous. Most tutorials show a simple ReAct loop calling a search API. Real enterprise agents need to handle authentication, maintain audit trails, respect RBAC boundaries, recover from failures gracefully, and operate within strict compliance frameworks. After building dozens of production agents at Citadel Cloud Management, we distilled the patterns into AgentForge, a structured curriculum that covers eight distinct categories of enterprise AI agents.

AgentForge is not a framework or SDK. It is a teaching curriculum with production-ready reference implementations. Each category includes architectural patterns, code templates, evaluation harnesses, and deployment guides. The full curriculum lives in the agentforge-ai-agent-curriculum repository on GitHub.

The core philosophy is that AI agents should be treated as software systems, not science experiments. That means version control, CI/CD pipelines, observability, and automated testing from day one. Every agent in the curriculum ships with a Terraform deployment module, a test harness, and a monitoring dashboard.

AgentForge Architecture Pattern

Every agent in the curriculum follows the same foundational architecture. This consistency makes it possible to compose agents into multi-agent systems and share infrastructure components like memory stores, tool registries, and evaluation pipelines.

# AgentForge base architecture (Python)
# Repository: https://github.com/kogunlowo123/agentforge-ai-agent-curriculum

from agentforge.core import Agent, ToolRegistry, Memory
from agentforge.guardrails import PolicyEngine
from agentforge.observability import TracingMiddleware

class EnterpriseAgent(Agent):
    def __init__(self, config: AgentConfig):
        self.tools = ToolRegistry.from_config(config.tools)
        self.memory = Memory(
            backend="dynamodb",
            table=config.memory_table,
            ttl_hours=config.memory_ttl
        )
        self.policy = PolicyEngine(
            rules=config.guardrail_rules,
            rbac_provider=config.rbac_provider
        )
        self.tracer = TracingMiddleware(
            service_name=config.agent_name,
            exporter="otlp"
        )

    async def execute(self, task: Task) -> AgentResult:
        # Pre-execution guardrail check
        self.policy.validate_task(task)

        # Load relevant context from memory
        context = await self.memory.recall(task.context_key)

        # Plan-Execute-Reflect loop
        plan = await self.plan(task, context)
        for step in plan.steps:
            self.policy.validate_action(step.action)
            result = await self.tools.execute(step.action)
            self.tracer.record_step(step, result)
            await self.memory.store(step.context_key, result)

        return self.synthesize(plan, results)

The key components are the ToolRegistry for managing tool access with RBAC, the Memory layer backed by DynamoDB or Redis for persistent context, the PolicyEngine that enforces guardrails before every action, and the TracingMiddleware that exports OpenTelemetry traces for every agent decision.

Building a DevOps Agent

The DevOps agent is the most requested category in the curriculum. It handles three primary workflows: CI/CD pipeline optimization, deployment orchestration, and incident triage. The agent connects to GitHub Actions, Terraform Cloud, and Kubernetes clusters to observe, plan, and execute operational tasks.

# DevOps Agent - Deployment Orchestration
# Repository: https://github.com/kogunlowo123/agentforge-ai-agent-curriculum

class DevOpsAgent(EnterpriseAgent):
    tools = [
        "github.list_pull_requests",
        "github.get_workflow_runs",
        "terraform.plan",
        "terraform.apply",
        "kubernetes.get_deployments",
        "kubernetes.rollout_status",
        "kubernetes.rollback",
        "datadog.get_metrics",
        "slack.post_message"
    ]

    async def handle_deployment(self, pr: PullRequest):
        # 1. Validate PR checks passed
        checks = await self.tools.github.get_check_runs(pr.sha)
        if not all(c.conclusion == "success" for c in checks):
            return AgentResult(action="blocked", reason="CI checks failing")

        # 2. Run Terraform plan for infrastructure changes
        plan = await self.tools.terraform.plan(workspace=pr.target_env)
        if plan.has_destructive_changes:
            await self.tools.slack.post_message(
                channel="#deployments",
                text=f"Destructive changes detected in {pr.url}. Manual approval required."
            )
            return AgentResult(action="awaiting_approval")

        # 3. Execute deployment with canary verification
        await self.tools.terraform.apply(workspace=pr.target_env)
        await self.tools.kubernetes.rollout_status(deployment=pr.service_name)

        # 4. Monitor error rate for 10 minutes post-deploy
        metrics = await self.tools.datadog.get_metrics(
            query=f"avg:http.error_rate{'{'}service:{pr.service_name}{'}'}",
            window="10m"
        )
        if metrics.avg > 0.05:
            await self.tools.kubernetes.rollback(deployment=pr.service_name)
            return AgentResult(action="rolled_back", reason="Error rate exceeded 5%")

The agent validates CI checks, runs Terraform plans, detects destructive changes requiring human approval, executes canary deployments, and automatically rolls back if error rates exceed thresholds. Every action is logged to the OpenTelemetry trace and persisted in the memory store for post-incident review.

Security Operations Agent

The security agent automates threat detection triage and compliance remediation. It ingests findings from AWS GuardDuty, Microsoft Defender for Cloud, and GCP Security Command Center, correlates them against known attack patterns, and executes automated remediation playbooks for low-risk findings while escalating high-risk detections to the security team.

A critical design constraint is that the security agent never has write access to production IAM policies. It can only propose changes through pull requests that require human approval. This separation of privilege is enforced at the PolicyEngine level and verified by integration tests in the CI pipeline.

The curriculum includes twelve pre-built remediation playbooks covering common findings like publicly accessible S3 buckets, overly permissive security groups, unencrypted EBS volumes, and expired SSL certificates. Each playbook is implemented as a Terraform module that the agent invokes through the ToolRegistry.

Multi-Agent Orchestration

Individual agents are useful, but the real power emerges when agents collaborate. AgentForge includes an orchestration layer that coordinates multiple agents through event-driven workflows. When the SRE agent detects a capacity issue, it can trigger the FinOps agent to evaluate cost implications and the DevOps agent to execute a scaling operation, all within a single traced workflow.

# Multi-Agent Orchestration via EventBridge
# Repository: https://github.com/kogunlowo123/agentforge-ai-agent-curriculum

orchestration:
  workflow: "incident-response"
  trigger:
    source: "sre-agent"
    event: "capacity-alert"
  steps:
    - agent: "sre-agent"
      action: "diagnose_capacity_issue"
      output: "diagnosis"
    - agent: "finops-agent"
      action: "evaluate_scaling_cost"
      input: "${diagnosis}"
      output: "cost_analysis"
    - agent: "devops-agent"
      action: "execute_scaling"
      input: "${diagnosis}"
      condition: "${cost_analysis.approved == true}"
    - agent: "documentation-agent"
      action: "update_runbook"
      input: "${diagnosis}, ${cost_analysis}"

The orchestration layer uses AWS EventBridge for event routing and Step Functions for workflow coordination. Each agent maintains its own execution context but shares a common memory namespace for the workflow, enabling downstream agents to access upstream results without tight coupling.

Agent Evaluation and Guardrails

Every agent in the curriculum ships with an evaluation harness that tests three dimensions: correctness (does the agent take the right actions), safety (does the agent respect guardrails and RBAC boundaries), and efficiency (does the agent minimize unnecessary tool calls and LLM invocations).

The guardrail system operates at three levels. Pre-execution guardrails validate that the requested task falls within the agent's authorized scope. Per-action guardrails check each tool invocation against RBAC policies and blast-radius limits. Post-execution guardrails verify that the cumulative effect of the agent's actions did not violate any invariants, such as the total infrastructure cost remaining within budget thresholds.

We use a combination of deterministic unit tests for tool integrations and LLM-as-judge evaluation for reasoning quality. The evaluation suite runs in CI on every commit, ensuring that model updates or prompt changes do not degrade agent behavior.

Getting Started with AgentForge

The curriculum is structured as a progressive learning path. Start with the DevOps agent module, which introduces all core concepts (tool registry, memory, guardrails, tracing) in a familiar domain. Then branch into your domain of interest. Each module includes a README with prerequisites, a Terraform deployment guide, example scenarios, and evaluation benchmarks.

Clone the repository and follow the quickstart guide to deploy your first agent in under 30 minutes. The only prerequisites are an AWS account, Python 3.11+, and Terraform 1.5+. All infrastructure is deployed via Terraform modules included in the repository.
