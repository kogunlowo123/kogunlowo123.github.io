What is the Model Context Protocol?

The Model Context Protocol (MCP) is an open standard that defines how AI models connect to external tools and data sources. Rather than embedding tool-calling logic inside application code, MCP provides a standardized server interface that any compatible AI client can discover and invoke. Think of it as a USB-C port for AI: a universal interface that any model can plug into.

An MCP server exposes three primitives: tools (actions the model can invoke), resources (data the model can read), and prompts (templated instructions for common tasks). When Claude connects to an MCP server, it automatically discovers the available tools and can invoke them during conversations with full context about parameter types and return values.

At Citadel Cloud Management, we built a suite of MCP servers that give Claude direct access to our infrastructure management toolchain. The result is an AI assistant that can query AWS resources, run Terraform plans, check security compliance, and draft remediation PRs, all within a natural conversation. The servers are available in our GitHub repositories.

Why MCP for Infrastructure Management

Infrastructure engineers spend enormous time on context-switching: jumping between the AWS console, Terraform files, monitoring dashboards, and Slack threads to diagnose and resolve issues. MCP servers collapse this context into a single conversational interface where Claude can access all these data sources simultaneously.

The key advantage over traditional chatbot integrations is that MCP is stateless and discoverable. You do not need to hard-code tool descriptions into your application. The MCP server self-describes its capabilities, and Claude adapts its behavior based on what tools are available. Add a new Terraform workspace tool and Claude immediately knows how to use it.

We measured a 40% reduction in mean time to resolution (MTTR) for infrastructure incidents after deploying MCP servers. Engineers ask Claude to diagnose the issue, and Claude pulls metrics from CloudWatch, checks recent deployments from Terraform Cloud, reviews pod logs from Kubernetes, and synthesizes a root cause analysis in seconds rather than the minutes it takes to manually navigate each system.

Building a Terraform MCP Server

The Terraform MCP server exposes five core tools that Claude can invoke to manage infrastructure. Here is the server implementation using the MCP Python SDK.

# Terraform MCP Server
# Repository: https://github.com/kogunlowo123/mcp-server-terraform

from mcp.server import Server
from mcp.types import Tool, TextContent
import subprocess
import json

server = Server("terraform-mcp")

@server.tool()
async def terraform_plan(workspace: str, target: str = None) -> str:
    """Run terraform plan on a workspace and return the plan summary."""
    cmd = ["terraform", "plan", "-no-color", "-json"]
    if target:
        cmd.extend(["-target", target])

    result = subprocess.run(
        cmd,
        cwd=f"/workspaces/{workspace}",
        capture_output=True, text=True, timeout=300
    )

    plan_data = parse_plan_output(result.stdout)
    return json.dumps({
        "workspace": workspace,
        "adds": plan_data["resource_changes"]["add"],
        "changes": plan_data["resource_changes"]["change"],
        "destroys": plan_data["resource_changes"]["destroy"],
        "summary": plan_data["summary"]
    })

@server.tool()
async def terraform_state_list(workspace: str) -> str:
    """List all resources in a Terraform workspace state."""
    result = subprocess.run(
        ["terraform", "state", "list"],
        cwd=f"/workspaces/{workspace}",
        capture_output=True, text=True
    )
    return result.stdout

@server.tool()
async def terraform_output(workspace: str, name: str = None) -> str:
    """Get outputs from a Terraform workspace."""
    cmd = ["terraform", "output", "-json"]
    if name:
        cmd.append(name)

    result = subprocess.run(
        cmd,
        cwd=f"/workspaces/{workspace}",
        capture_output=True, text=True
    )
    return result.stdout

@server.tool()
async def terraform_show_resource(workspace: str, address: str) -> str:
    """Show the current state of a specific Terraform resource."""
    result = subprocess.run(
        ["terraform", "state", "show", address],
        cwd=f"/workspaces/{workspace}",
        capture_output=True, text=True
    )
    return result.stdout

Notice that the server does not expose terraform apply as a direct tool. Applies require an approval gate, which we implement through a separate workflow described in the guardrails section below.

AWS MCP Server: Read and Remediate

The AWS MCP server provides read access to AWS resources and the ability to generate remediation Terraform code. It uses boto3 with an IAM role that has read-only access to the target account, plus write access to a dedicated Git repository for remediation PRs.

# AWS MCP Server - Resource Discovery Tool
@server.tool()
async def aws_describe_security_groups(
    vpc_id: str,
    filter_public: bool = True
) -> str:
    """List security groups, optionally filtering for public access rules."""
    ec2 = boto3.client("ec2")
    response = ec2.describe_security_groups(
        Filters=[{"Name": "vpc-id", "Values": [vpc_id]}]
    )

    groups = []
    for sg in response["SecurityGroups"]:
        public_rules = [
            rule for rule in sg["IpPermissions"]
            if any(ip["CidrIp"] == "0.0.0.0/0" for ip in rule.get("IpRanges", []))
        ]
        if filter_public and not public_rules:
            continue
        groups.append({
            "id": sg["GroupId"],
            "name": sg["GroupName"],
            "public_rules": len(public_rules),
            "description": sg["Description"]
        })

    return json.dumps(groups, indent=2)

When Claude identifies a misconfigured resource, it generates a Terraform remediation plan and opens a pull request in the infrastructure repository. The engineer reviews and approves the PR, and the CI/CD pipeline applies the change. This maintains the human-in-the-loop requirement while dramatically accelerating the remediation cycle.

Guardrails and Approval Gates

Giving an AI model access to cloud infrastructure requires robust guardrails. Our MCP servers enforce three layers of protection. First, IAM-level restrictions ensure the MCP server's credentials can only perform read operations on production resources. Second, tool-level guardrails validate every invocation against an allowlist of permitted operations. Third, workflow-level approval gates require human sign-off before any state-changing operation executes.

The approval gate works through GitHub pull requests. When Claude determines that a Terraform apply is needed, it creates a PR with the planned changes, tags the appropriate reviewers, and includes a summary of why the change is necessary. Only after a human approves and merges the PR does the CI pipeline execute the apply.

Audit Trail and Compliance

Every MCP server invocation is logged to CloudWatch Logs with the full request and response payload, the identity of the user who initiated the conversation, and a correlation ID that links the MCP call back to the Claude conversation. This audit trail satisfies SOC 2 Type II requirements for access logging and provides forensic capability for incident investigation.

We also export MCP server metrics to Datadog, tracking tool invocation counts, latency percentiles, error rates, and the distribution of tool usage across users. This telemetry helps us identify which tools are most valuable and where to invest in improving the user experience.

Deploying MCP Servers in Production

Our MCP servers run as containerized services on ECS Fargate, deployed via the Terraform modules from our open-source library. Each server runs in its own task with a dedicated IAM role, and inter-server communication is encrypted with mutual TLS via the service mesh. The deployment is fully automated through GitHub Actions and requires zero manual intervention.
