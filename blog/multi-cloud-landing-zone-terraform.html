The Multi-Cloud Landing Zone Problem

Every enterprise that operates across AWS, Azure, and GCP faces the same challenge: each cloud provider has its own landing zone framework (AWS Control Tower, Azure Landing Zones, GCP Cloud Foundation Toolkit), but none of them talk to each other. You end up with three separate governance models, three networking architectures, three identity systems, and three sets of compliance controls. Drift between them is inevitable, and auditing consistency is a full-time job.

Our multi-cloud-landing-zone repository solves this by providing a single Terraform codebase that deploys a consistent landing zone across all three providers. The configuration is parameterized at the top level, and cloud-specific implementation details are abstracted behind a unified module interface.

Unified Architecture Design

The landing zone architecture is built on four pillars: organization structure, networking, identity federation, and security baseline. Each pillar has a unified interface that maps to provider-specific implementations underneath.

Account and Project Structure

The landing zone creates a standardized hierarchy across all three providers. At the top level, organizational units map to management groups map to folders. Each environment (development, staging, production) gets its own OU/management group/folder with corresponding policy guardrails.

# Unified organization structure definition
# Repository: https://github.com/kogunlowo123/multi-cloud-landing-zone

organization = {
  name = "citadel-corp"

  units = {
    platform = {
      accounts = {
        networking  = { budget = 5000 }
        security    = { budget = 3000 }
        shared-svcs = { budget = 4000 }
      }
    }
    workloads = {
      environments = {
        dev = {
          accounts_per_team = 1
          allow_public_access = true
          budget_per_account  = 2000
        }
        staging = {
          accounts_per_team = 1
          allow_public_access = false
          budget_per_account  = 5000
        }
        production = {
          accounts_per_team = 2  # primary + DR
          allow_public_access = false
          budget_per_account  = 20000
          require_mfa         = true
          require_encryption  = true
        }
      }
    }
    sandbox = {
      auto_nuke_after_days = 7
      budget_per_account   = 500
    }
  }
}

This single configuration generates AWS accounts via Organizations, Azure subscriptions via Management Groups, and GCP projects via Folders. Each account/subscription/project inherits the governance policies defined at its level in the hierarchy.

Cross-Cloud Networking

The networking pillar deploys a hub-spoke topology within each cloud and establishes encrypted tunnels between the hubs. AWS Transit Gateway connects to Azure Virtual WAN via IPSec VPN, and both connect to GCP through Cloud Interconnect or VPN. All inter-cloud traffic traverses these encrypted tunnels, and each spoke network is isolated with firewall rules that default to deny-all.

CIDR allocation is managed centrally to prevent overlap. The landing zone reserves non-overlapping ranges for each cloud and each environment, with room for growth. Our IPAM module tracks allocations in a DynamoDB table and validates new requests against existing assignments.

# Cross-cloud network configuration
networking = {
  cidr_allocation = {
    aws   = "10.0.0.0/12"    # 10.0.0.0 - 10.15.255.255
    azure = "10.16.0.0/12"   # 10.16.0.0 - 10.31.255.255
    gcp   = "10.32.0.0/12"   # 10.32.0.0 - 10.47.255.255
  }

  cross_cloud_vpn = {
    aws_to_azure = { enabled = true, bandwidth = "1Gbps" }
    aws_to_gcp   = { enabled = true, bandwidth = "1Gbps" }
    azure_to_gcp = { enabled = true, bandwidth = "1Gbps" }
  }

  dns = {
    primary_zone   = "citadel-corp.internal"
    forwarding     = true  # Cross-cloud DNS resolution
  }
}

Unified Governance Layer

Governance policies are defined once in a cloud-agnostic format and compiled to provider-specific policy languages. For example, a rule that says "all storage must be encrypted with customer-managed keys" translates to an AWS SCP that denies s3:CreateBucket without encryption, an Azure Policy that audits storage accounts without CMK encryption, and a GCP Organization Policy constraint that requires CMEK for Cloud Storage.

The governance module currently supports 45 policies covering encryption, public access, network restrictions, tagging requirements, and resource location constraints. Each policy includes a severity level, a remediation guide, and a Terraform module that can fix non-compliant resources automatically.

Terraform Implementation

The landing zone is deployed in phases to handle provider-specific bootstrapping requirements. Phase one creates the organization structure and IAM roles. Phase two deploys networking and VPN tunnels. Phase three enables security services and monitoring. Phase four creates workload accounts and applies governance policies.

# Deploy the landing zone
# Repository: https://github.com/kogunlowo123/multi-cloud-landing-zone

module "landing_zone" {
  source = "./modules/multi-cloud-landing-zone"

  organization = local.organization
  networking   = local.networking
  governance   = local.governance

  providers = {
    aws   = aws.management
    azurerm = azurerm.management
    google  = google.organization
  }

  # Phase control - deploy incrementally
  enable_phase_1_org       = true
  enable_phase_2_network   = true
  enable_phase_3_security  = true
  enable_phase_4_workloads = true
}

output "aws_account_ids" {
  value = module.landing_zone.aws_accounts
}

output "azure_subscription_ids" {
  value = module.landing_zone.azure_subscriptions
}

output "gcp_project_ids" {
  value = module.landing_zone.gcp_projects
}

Day-Two Operations

A landing zone is only as good as its day-two operations. The repository includes a drift detection pipeline that runs hourly, comparing the actual state of each cloud against the Terraform state. Any drift triggers a Slack notification with a diff and a one-click remediation button that creates a PR to bring the environment back into compliance.

New workload accounts are provisioned through a self-service portal backed by a GitHub Actions workflow. Teams submit a YAML request, the workflow validates it against organizational policies, creates the accounts across all requested clouds, and provisions the networking and security baseline automatically. Average provisioning time from request to ready-to-use is under 15 minutes.
