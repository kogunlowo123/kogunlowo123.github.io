Why We Open-Sourced Our Module Library

Over the past three years at Citadel Cloud Management, we accumulated a library of Terraform modules that power production infrastructure across dozens of enterprise clients. Each module had been battle-tested through hundreds of deployments, refined through incident post-mortems, and hardened with security best practices. We made the decision to open-source the entire library because we believe the infrastructure community benefits more from production-tested patterns than from yet another "getting started" tutorial.

The library is available across individual repositories on GitHub, each following a consistent naming convention: terraform-{provider}-{resource}. Every module includes comprehensive documentation, example configurations, automated tests, and an interactive architecture diagram.

Module Design Standards

Consistency across 70+ modules requires strict design standards. Every module adheres to the following conventions.

All modules use a flat variable structure with sensible defaults. Required variables have no defaults; optional variables always have a default value that represents the most secure configuration. For example, encryption is always enabled by default, public access is always disabled by default, and logging is always enabled by default.

# Module structure standard
# Every module follows this directory layout

terraform-aws-eks-cluster/
  main.tf              # Primary resource definitions
  variables.tf         # Input variables with descriptions and validation
  outputs.tf           # Output values for downstream consumption
  versions.tf          # Provider and Terraform version constraints
  locals.tf            # Computed values and naming conventions
  data.tf              # Data sources for existing resources
  README.md            # Auto-generated documentation
  diagram.html         # Interactive architecture diagram
  examples/
    basic/             # Minimal configuration example
    complete/          # Full-featured configuration example
  tests/
    basic_test.go      # Terratest for basic example
    complete_test.go   # Terratest for complete example
    unit.tftest.hcl    # Native Terraform test

Every variable includes a description, a type constraint, and a validation block where applicable. This ensures that users get meaningful error messages before terraform plan ever runs.

# Example: variable validation in terraform-aws-eks-cluster
variable "cluster_version" {
  description = "Kubernetes version for the EKS cluster"
  type        = string
  default     = "1.29"

  validation {
    condition     = contains(["1.27", "1.28", "1.29", "1.30"], var.cluster_version)
    error_message = "Cluster version must be one of: 1.27, 1.28, 1.29, 1.30."
  }
}

variable "node_instance_types" {
  description = "List of EC2 instance types for the managed node group"
  type        = list(string)
  default     = ["m6i.large", "m6i.xlarge"]

  validation {
    condition     = length(var.node_instance_types) > 0
    error_message = "At least one instance type must be specified."
  }
}

Automated Testing with Terratest and Native Tests

Every module ships with two layers of tests. Terratest integration tests deploy real infrastructure, validate the outputs, and tear it down. Native Terraform tests (.tftest.hcl) validate plan-time behavior without deploying resources, making them fast enough to run on every pull request.

Our CI pipeline runs native tests on every PR and Terratest integration tests on a nightly schedule against dedicated test accounts in each cloud provider. Test accounts are provisioned with the minimum IAM permissions required for each module, ensuring that the tests validate the documented permission requirements.

# Native Terraform test example (unit.tftest.hcl)
run "validates_cluster_name_length" {
  command = plan

  variables {
    cluster_name = "this-name-is-way-too-long-for-eks-and-should-fail-validation"
    vpc_id       = "vpc-12345678"
    subnet_ids   = ["subnet-abc", "subnet-def"]
  }

  expect_failures = [var.cluster_name]
}

run "creates_cluster_with_defaults" {
  command = plan

  variables {
    cluster_name = "test-cluster"
    vpc_id       = "vpc-12345678"
    subnet_ids   = ["subnet-abc", "subnet-def"]
  }

  assert {
    condition     = aws_eks_cluster.this.version == "1.29"
    error_message = "Default cluster version should be 1.29"
  }
}

Documentation Generation Pipeline

Documentation is generated automatically from the Terraform code using terraform-docs and a custom pipeline that produces the README.md, the interactive architecture diagram, and the Terraform Registry metadata. When a module is updated, the CI pipeline regenerates all documentation and commits it alongside the code change.

The interactive architecture diagrams are generated as standalone HTML files using our architecture-diagrams-as-code toolchain. Each diagram shows the resources created by the module, their relationships, and the data flows between them. Users can click on any resource to see its Terraform configuration.

Semantic Versioning and Release Automation

Every module follows strict semantic versioning. Breaking changes (removed variables, changed defaults, provider version bumps) increment the major version. New features and optional variables increment the minor version. Bug fixes and documentation updates increment the patch version.

Releases are automated through GitHub Actions. When a PR is merged to main, the CI pipeline analyzes the commit messages (following Conventional Commits), determines the appropriate version bump, generates a changelog, creates a GitHub release, and publishes to the Terraform Registry. The entire process requires zero manual intervention.

Using the Modules in Your Projects

All modules are available directly from GitHub. Reference them in your Terraform configurations using the GitHub source with a version pin.

# Using a module from the library
module "eks_cluster" {
  source  = "github.com/kogunlowo123/terraform-aws-eks-cluster?ref=v3.2.1"

  cluster_name    = "production"
  cluster_version = "1.29"
  vpc_id          = module.vpc.vpc_id
  subnet_ids      = module.vpc.private_subnet_ids

  node_groups = {
    general = {
      instance_types = ["m6i.xlarge"]
      min_size       = 3
      max_size       = 10
      desired_size   = 5
    }
  }

  enable_cluster_autoscaler = true
  enable_metrics_server     = true
  enable_aws_load_balancer  = true
}

module "vpc" {
  source  = "github.com/kogunlowo123/terraform-aws-vpc-network?ref=v2.8.0"

  vpc_name   = "production"
  cidr_block = "10.0.0.0/16"
  azs        = ["us-east-1a", "us-east-1b", "us-east-1c"]

  enable_nat_gateway   = true
  enable_flow_logs     = true
  enable_transit_gateway = false
}

We recommend pinning to a specific version tag and using Dependabot or Renovate to automate version updates. Each release includes a detailed changelog that documents any migration steps required for breaking changes.
